# ADR-0001 — 实时行情推送架构（服务端代理 Massive 股票 WebSocket）

日期：2026-02-09  
状态：Accepted（2026-03-03 更新）

## 背景

- `PRD-0001` 当前仅要求股票实时更新能力。
- Massive 提供 WebSocket，但浏览器直连会带来：
  - API Key 暴露风险
  - 订阅规模不可控
  - 多用户场景下连接治理困难

## 决策

采用“服务端代理 + 前端订阅”的模式：

1. 服务端进程统一连接 Massive 股票 WS 上游。  
2. API 提供 `WS /api/v1/market-data/stream` 给前端。  
3. Redis Pub/Sub 负责跨实例广播。  
4. WS 异常时自动降级到 REST 轮询。  

## 备选方案

### 方案 A：前端直连 Massive WS

- 优点：实现简单、理论延迟更低。
- 缺点：密钥暴露、订阅不可控、前端状态机复杂。

### 方案 B：服务端代理（采纳）

- 优点：密钥安全、统一鉴权与限流、可观测性更好。
- 缺点：需要维护长连接与重连策略。

### 方案 C：仅 REST 轮询

- 优点：实现最简单。
- 缺点：实时性差，成本与请求压力更高。

## 影响

- API 负责鉴权、订阅管理、扇出与错误语义。
- 前端负责连接状态展示与降级恢复。
- Redis 同时承担实时广播通道。

## 边界说明

- 本 ADR 当前只覆盖股票实时链路。
- 非股票扩展实时能力已从当前范围移除；未来需要时重新评审并新增 ADR。
